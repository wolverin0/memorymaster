**Position:** The skill is plausible for one repo, but unsafe as a 15+ project generator because too many steps depend on environment quirks, inferred state, and non-idempotent regeneration.

**Key Problems:**
- **Wrapper drift is already visible.** `CLAUDE.md` is supposed to be a thin wrapper around `AGENTS.md`, but it also inlines a full GitNexus block. That means regeneration can duplicate or desynchronize the very sections it claims to preserve. “Preserve markers” is underspecified: preserve only managed blocks, or preserve any matching comment pair found anywhere?
- **Rollback is probably incomplete.** Backing up existing files is not enough if the run also creates new files (`GEMINI.md`, `.gemini/settings.json`, fresh wrappers, maybe rules). A rollback that only restores prior files still leaves generated artifacts behind and does not restore pre-run directory state.
- **`--dry-run` is ambiguous.** If dry-run still explores the repo, reads MCP, derives scope from git, and renders outputs, it can fail on exactly the same filesystem/tooling issues as a real run. If it skips those steps, the preview is not faithful. The contract needs to define which side effects are suppressed and which validations still run.
- **Codebase exploration is brittle at scale.** GSD-style repo scanning will hit `node_modules`, `.pytest_cache`, generated dirs, worktrees, vendor clones, and huge monorepos. We already saw `EPERM`/scandir-style failures from Gemini on this repo. Without hard ignore rules and size/time budgets, “explore first” becomes a failure source.
- **MemoryMaster is treated as required, but availability is not guaranteed.** Cross-project rollout will hit repos where MCP is disconnected, workspace-scoped, or empty. The skill needs explicit fallback behavior, otherwise generation quality silently degrades or the run aborts.
- **Scope derivation from git remote is not stable enough.** Forks, renamed repos, monorepos, multiple remotes, detached clones, and local-only repos can all map to the wrong scope. Worse, two unrelated repos can collapse to the same normalized slug and contaminate memory recall.
- **`--force` is dangerous without a merge policy.** If wrappers or canonical files contain hand-edited local notes, a force rewrite can erase valid project-specific instructions. The skill needs a managed/unmanaged section model, not whole-file replacement with backup as the safety net.

**Bottom line:** the main weakness is not content generation; it is state management. Until idempotency, rollback semantics, ignore rules, and scope derivation are made precise, this will break unevenly across a multi-project fleet.
