# Research Notes: Unified Agent Instruction Architecture Evaluation

## 1. 4-Layer Memory Governance Model
**Status: Correct and Highly Effective.**
- **Global (`~/.claude/CLAUDE.md`):** Successfully abstracts "Agent Personality" and "Work Habits" (STOP rules, ARG/AFIP context, multi-agent roles). This prevents redundancy across repos.
- **Project (`AGENTS.md` + `CLAUDE.md` wrapper):** `AGENTS.md` serves as the "Canonical Source of Truth" for all agents (Claude, Codex, Gemini). Using `@AGENTS.md` imports in `CLAUDE.md` is a masterstroke for maintenance.
- **Path-specific (`.claude/rules/*.md`):** Correctly handles domain-specific context (e.g., frontend vs. backend) without polluting the root instructions.
- **Ephemeral (MemoryMaster):** Captures the "living" state of the project (bugs, decisions) that is too volatile for static Markdown files.
- **Verdict:** This hierarchy solves the "Context Bloat" problem while maintaining high specificity.

## 2. Project-Setup Skill Analysis
**Status: Robust but needs "Multi-Agent" hardening.**
- **Strengths:** Automated stack detection and command verification (`ls`, `cat`, `grep`) ensure instructions aren't just "hallucinated placeholders." The <200 lines constraint is critical for LLM adherence.
- **Weaknesses:** While it generates `CLAUDE.md`, it assumes Gemini and Codex will just "find" `AGENTS.md`. 
- **Recommendation:** The skill should optionally generate a tiny `GEMINI.md` or `.codex/AGENTS.md` that explicitly points to the root `AGENTS.md` to ensure zero-config rollout for those CLI tools.

## 3. MemoryMaster Integration
**Status: Properly Represented.**
- **Implementation:** The "Recall Hook" (Read) and "Auto-Dream" (Write) flow ensures memory is a passive background utility rather than a manual chore.
- **Wiki Pattern:** The "Compiled Truth (Above the line) + Timeline (Below the line)" exactly mirrors the **Karpathy/Farza "LLM Wiki" pattern**. It solves the problem of "RAG Noise" by forcing a synthesis step (Steward cycle).
- **Self-Correction:** `AGENTS.md` in the `memorymaster` repo is a great example of "Recursive Memory"—the agent uses the memory system to maintain the memory system.

## 4. Alignment with Industry Patterns
- **Karpathy/Farza/GBrain:** The architecture aligns with the shift from "Instruction Prompting" to "Context Retrieval." Instead of 2000-line prompt files, it uses a "Kernel" (Global) + "Drivers" (Project) + "Virtual Memory" (MemoryMaster).
- **Single Source of Truth:** `new 1.txt` correctly identified that maintaining three separate instruction files (Claude/Gemini/Codex) is a "maintenance nightmare." The `AGENTS.md` pattern is the correct industry-standard approach.

## 5. Missing or Wrong Elements
- **GitNexus Bloat:** The `AGENTS.md` in the `memorymaster` repo has a very large GitNexus section. If this is copied to all 15+ projects, it will eat context. This should be moved to a `.claude/rules/gitnexus.md` or kept in the global layer if the tools are global.
- **Verification Loop:** The global `CLAUDE.md` forbids "This should work." However, the `project-setup` skill doesn't explicitly add a "Verification" section to the generated `AGENTS.md` beyond a generic "Testing" table.
- **Steward Visibility:** The 6-hour Steward cycle is a "black box." Agents should have a way to see *when* the last re-synthesis happened to trust the Wiki's "Compiled Truth."

## Conclusion for Debate
The architecture is **90% correct** for a 15+ project rollout. The move to `AGENTS.md` as a shared canonical core is the winning move. The primary risk is **"Instruction Drift"** if the `project-setup` skill isn't run periodically to refresh the "Real Commands" section.
