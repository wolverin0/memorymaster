# MemoryMaster Security Audit - 2026-05-11

Scope: `memorymaster/mcp_server.py`, `memorymaster/dashboard.py`, attempted `memorymaster/dashboard_helpers*.py`, `memorymaster/service.py`, and `memorymaster/llm_provider.py`.

No production code was modified. Evidence was read from GitHub `wolverin0/memorymaster` `main` plus the indexed local symbols for line grounding. `memorymaster/dashboard_helpers.py` and `memorymaster/dashboard_helpers_extra.py` were not present on `main` via the GitHub contents API, so they are recorded as absent in the threat model.

## Executive Summary

The highest risk area is the unauthenticated local dashboard: GET routes expose claims, events, review queues, observability, and operator logs, while POST routes mutate claims, proposals, and start or stop the operator process. This is acceptable only if the dashboard is strictly loopback and trusted-user only; it becomes high risk if bound to a shared interface, proxied, or reached by a local web origin through CSRF-style requests.

MCP tool entrypoints have broader power than a normal user-facing API. Tools accept caller-controlled database paths and workspace roots, and several mutating tools operate without role or tenant context. The service layer does redact sensitive ingests and filters sensitive reads by default, but cross-scope tools such as `federated_query` and raw transcript tools need stronger access-control boundaries.

No Critical findings were identified from code paths read. The audit found complete STRIDE and OWASP 2021 coverage with concrete findings or explicit no-threat justifications in `docs/security-audit/findings.md`.
