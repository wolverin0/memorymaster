## What this measures

Each call to `memorymaster.context_hook.recall()` now emits structured
latency events to the shared hook log (`~/.memorymaster/hook_state/hook.log`
via `memorymaster.hook_log.log_hook`). The aggregator
(`scripts/agg_recall_latency.py`) reads the log and prints one row per stream
with N / p50 / p99 / mean, all in milliseconds.

Streams instrumented:

| Stream            | What it covers                                                                    |
|-------------------|-----------------------------------------------------------------------------------|
| `fts5`            | Per-token `svc.query_rows()` fan-out PLUS the raw-prompt fallback                 |
| `entity_fanout`   | `_entity_fanout_claim_ids` + row materialisation (only when `W_ENTITY>0` or empty)|
| `bm25_rescore`    | Per-field BM25 corpus stats + scoring (only when `LEXICAL_BM25` is on)            |
| `vector_fallback` | `_apply_vector_fallback` (only when `RECALL_VECTOR_FALLBACK=1` AND Qdrant URL set)|
| `verbatim`        | MemPalace-style stream (only when `RECALL_VERBATIM=1`)                            |
| `rank_and_build`  | `_relevance`/RRF sort + output-budget loop (skipped on the empty-rows early exit) |
| `total`           | Wall clock of the whole `recall()` call                                           |

All timers use `time.perf_counter()`, NOT `time.monotonic()`, because
claim 11848 documents a Windows timer flake where `monotonic()` can return
non-monotonic values across a clock-sync boundary. `perf_counter` is the
stdlib-recommended short-interval timer on all platforms.

## Outliers (>100 ms total)

Only the 30-prompt set contains a single outlier — call `#0` at 337 ms. On
the 100-prompt run the same prompt `#0` cost 322 ms; every subsequent call
stayed under 100 ms.

| rank | call | total_ms | prompt (first 80 chars)                                                              |
|------|------|----------|--------------------------------------------------------------------------------------|
| 1    | #0   | 322.8    | `hace C, no tenemos nada que perder, go.`                                            |
| 2    | #50  | 90.6     | `español ahora; veo que codex tien econfigurado el memorymaster pero era el viejo, h`|
| 3    | #80  | 84.3     | `y pero depende de claude.. no podemos forzarlo de alguna forma con algun stop hook?`|

**Cause of the #0 outlier:** cold-import cost. The first `recall()` call
pays for import of `memorymaster.service` (→ storage, FTS5, postgres_store,
tokenizer, ...). The `fts5` phase absorbs most of it because the first
`svc.query_rows()` triggers SQLite's WAL setup and the FTS5 compiled query.
On every subsequent call the imports are cached, and `fts5` drops back to
its stable p50 of ~52 ms.

The #50 / #80 prompts are genuine long-prompt cases — many tokens to search
FTS5 with, so the per-token loop makes more queries. Still within
"interactive" budget.

## Headline numbers

- **p50 total** = **53 ms** (100-prompt set). `fts5` dominates.
- **p99 total** = **91 ms** excluding cold-start.
- `bm25_rescore` and `rank_and_build` together contribute well under 1 ms
  at p99 — they are not worth optimising further until the FTS5 stream is
  addressed.
- Per-stream overhead of the timer itself (measured in isolation with 7
  back-to-back `perf_counter()` pairs × 10 000 iterations): **0.5 µs
  avg, 2.3 µs max** per call. Well below the "<1 ms in aggregate"
  non-negotiable rule from the spec.

## Reproducing

```bash
# Run recall against the 100-prompt set and dump latency events to a fresh log
python - <<'PY'
import json, tempfile
from pathlib import Path
tmp = Path(tempfile.gettempdir()) / 'recall-latency-2026-04-23'; tmp.mkdir(parents=True, exist_ok=True)
log = tmp / 'hook.log'; log.unlink(missing_ok=True)
import memorymaster.hook_log as hl
hl._STATE_DIR = tmp; hl._LOG_FILE = log
from memorymaster.context_hook import recall
db = 'memorymaster.db'  # adjust
with open('artifacts/real-prompts-100.jsonl', encoding='utf-8') as fh:
    for line in fh:
        t = json.loads(line.strip()).get('text','')
        if t: recall(t, db_path=db, skip_qdrant=True)
print('log:', log)
PY

# Aggregate
python scripts/agg_recall_latency.py <path-from-above>
```

## Next steps (NOT done in this roadmap item)

- FTS5 is the single fat stream. If we want sub-10 ms p99 we need to look
  at `svc.query_rows` — likely candidates are the per-token SQLite plan
  cache and a narrower `LIMIT` on the first token.
- Adding a percentile-buckets histogram to the aggregator (e.g. 10/50/90/99)
  would help catch multi-modal distributions — but out of scope for 5.1.
