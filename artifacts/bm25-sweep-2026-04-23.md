# BM25 k1/b sweep — 2026-04-23

Autoresearch Wave follow-up to the tokenizer v2 work (commit `7d80d83`).
Goal: tune lexical re-ranker hyperparameters to improve precision@5 on the
30-prompt eval (`artifacts/real-prompts.jsonl`).

## Summary

| Metric | Before (tokenizer v2, overlap-based lexical) | After (BM25 k1=1.2, b=0.25) | delta |
|--------|---------------------------------------------|-----------------------------|-------|
| precision@5 | 0.280 | **0.393** | **+0.113** |
| MAP@5       | 0.424 | **0.532** | **+0.108** |
| non-empty/30| 28     | **28**    | 0      |

**Ship gate**: require `delta p@5 >= 0.02` AND `delta MAP@5 >= 0.01` AND
`non-empty >= 28`. All three satisfied. Shipped.

## Why this works

SQLite FTS5's built-in `bm25()` hard-codes k1=1.2, b=0.75 and exposes only
per-column weights. It was being called as `bm25(claims_fts)` inside
`_storage_read.list_claims` purely to ORDER BY the SQL result — the value
was never propagated to the downstream re-ranker.

Meanwhile the re-ranker in `context_hook._relevance` uses a
`lexical_score` field produced by `retrieval._lexical_score`, which is
**not BM25 at all** — it is a token recall/precision/phrase/prefix blend.
Replacing that blend with a proper BM25(k1, b) score, computed in Python
over the candidate shortlist, is what produced the lift.

In other words: this wasn't really tuning FTS5's BM25 — it was replacing
an overlap-style lexical scorer with a real BM25 scorer and *then* tuning
its k1/b hyperparameters.

## Method

1. **Candidate fanout unchanged.** The same per-token FTS5 MATCH fanout
   the hook has always used — so candidate recall (non-empty rate) is
   identical to the baseline by construction.

2. **BM25 corpus stats** are built over the candidate set per call
   (~392 docs total across the 30 prompts; `avg_dl=49.1` tokens;
   vocab=6214). This is read-only-cheap — no extra DB traffic past the
   fetches the hook already does. Docs are tokenised with the existing
   `recall_tokenizer._candidate_tokens` so query-side and doc-side
   agree.

3. **Okapi BM25** (Robertson/Spärck Jones non-negative IDF variant):
   ```
   idf(q) = log( (N - n_q + 0.5) / (n_q + 0.5) + 1 )
   score  = Σ idf(q) · ( f(q,d) · (k1 + 1) )
                       / ( f(q,d) + k1·(1 - b + b·|d|/avgdl) )
   ```

4. **Grid**: `k1 ∈ {0.8, 1.2, 1.6, 2.0, 2.5}`, `b ∈ {0.0, 0.25, 0.5, 0.75, 1.0}`.
   25 combos, one-shot candidate fetch reused across the grid.

5. **Re-rank** uses the same context_hook weights
   `(matches=0.3, phrase=0.3, all=0.2, lexical=0.1, confidence=0.1,
   freshness=0, vector=0)` with `lexical` substituted by BM25(k1,b).

## Observations

- **p@5 plateaus at 0.393 across many combos.** The re-rank formula gives
  BM25 only `w_lexical=0.1`; the heavy lifters are `matches`, `phrase`,
  `all_present`. BM25 differentiates tie-breaks, not the top-line.
  A future follow-up could try raising `w_lexical` now that the lexical
  signal is sharper.
- **High `b` (≥0.75) hurts.** Length normalisation penalises longer
  claims, but many of our relevant claims ARE the longer, more
  information-dense ones (full lessons-learned paragraphs). Shipping with
  `b=0.25` preserves most short-doc boost without over-penalising the
  info-dense long docs.
- **k1 is essentially flat in [0.8, 2.5]** at low b. Shipping with
  `k1=1.2` (the canonical default) for principle.
- **Non-empty rate preserved at 28/30.** Two prompts have zero candidate
  hits regardless of scorer — that's a recall-side issue, not rank-side.

## Winner

**k1 = 1.2, b = 0.25.** Overrideable via env vars:

- `MEMORYMASTER_BM25_K1` (default `1.2`)
- `MEMORYMASTER_BM25_B` (default `0.25`)
- `MEMORYMASTER_LEXICAL_BM25` (default `1`; set to `0` to fall back to
  the previous overlap-based scorer)

## Wiring

- **Harness**: `scripts/eval_bm25_sweep.py` — 25-combo grid, single-point
  mode, JSON output. Read-only against the DB.
- **Prod change**: `memorymaster/context_hook.py::recall` now computes
  BM25 in-place over the retrieved candidate set and feeds it as the
  `lexical` term in `_relevance`. Fully backwards compatible via
  `MEMORYMASTER_LEXICAL_BM25=0`.
- **No deps added.** Pure stdlib (`math.log` + dict counts).
